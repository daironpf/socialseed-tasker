# Issue #07: Implement FastAPI REST API for Agent-to-Agent Communication

## Description

Build the web API entrypoint in `src/socialseed_tasker/entrypoints/web_api/` using **FastAPI**. This API serves as the "Sidecar" that allows external AI Agents to interact with the tasker via REST endpoints. It must generate a complete `openapi.json` for AI-discovery.

### Requirements

#### API Structure

**`app.py`** - FastAPI application factory
- Create FastAPI app with proper metadata (title, description, version, license)
- Configure CORS for browser access
- Register all routers
- Set up dependency injection for repositories
- Configure OpenAPI schema customization for AI agents

**`routes.py`** - API route definitions (can be split into multiple files as needed)

#### Required Endpoints

**Issues:**
- `POST /api/v1/issues` - Create a new issue
- `GET /api/v1/issues` - List issues with query params (status, component, labels, page, limit)
- `GET /api/v1/issues/{issue_id}` - Get issue details
- `PATCH /api/v1/issues/{issue_id}` - Update an issue
- `DELETE /api/v1/issues/{issue_id}` - Delete an issue
- `POST /api/v1/issues/{issue_id}/close` - Close an issue

**Dependencies:**
- `POST /api/v1/issues/{issue_id}/dependencies` - Add dependency
- `DELETE /api/v1/issues/{issue_id}/dependencies/{depends_on_id}` - Remove dependency
- `GET /api/v1/issues/{issue_id}/dependencies` - List dependencies
- `GET /api/v1/issues/{issue_id}/dependents` - List dependents
- `GET /api/v1/issues/{issue_id}/dependency-chain` - Get full transitive chain
- `GET /api/v1/blocked-issues` - Get all blocked issues

**Components:**
- `POST /api/v1/components` - Create a component
- `GET /api/v1/components` - List components
- `GET /api/v1/components/{component_id}` - Get component details

**Health & Discovery:**
- `GET /health` - Health check endpoint
- `GET /api/v1/openapi` - Custom OpenAPI endpoint with AI-friendly descriptions
- `GET /openapi.json` - Standard OpenAPI schema (auto-generated by FastAPI)

#### Request/Response Requirements
- All request bodies must use **Pydantic v2** models
- All responses must be wrapped in a consistent envelope:
  ```json
  {
    "data": { ... },
    "error": null,
    "meta": { "timestamp": "...", "request_id": "..." }
  }
  ```
- Proper HTTP status codes (201 for created, 204 for deleted, 400 for validation, 404 for not found, 409 for conflicts)
- Pagination support for list endpoints (page, limit, total, has_next, has_prev)
- Filtering support via query parameters

#### OpenAPI Enhancements for AI Discovery
- Add detailed descriptions to all endpoints explaining their purpose
- Include example request/response bodies in the schema
- Add `x-ai-agent` custom metadata fields where helpful
- Ensure all endpoints are properly tagged and grouped
- Generate schema that external agents can parse to understand available operations

#### Error Handling
- Global exception handler for unhandled errors (500)
- Validation error handler returning 422 with field-level details
- Custom exception handlers for domain errors (IssueNotFoundError -> 404, CircularDependencyError -> 409)
- Consistent error response format:
  ```json
  {
    "data": null,
    "error": {
      "code": "ISSUE_NOT_FOUND",
      "message": "Issue with id 'xyz' was not found",
      "details": {}
    }
  }
  ```

### Requirements
- All API code goes in `entrypoints/web_api/`
- Must use dependency injection via FastAPI's `Depends()`
- No business logic in routes - delegate to core actions
- API must be stateless (no session storage)
- Support async where appropriate (database calls)

### Business Value

The REST API enables AI agents (both internal and external) to programmatically interact with the task management system. The OpenAPI schema serves as a machine-readable contract that agents can parse to discover available operations, making the system self-documenting and agent-friendly.
