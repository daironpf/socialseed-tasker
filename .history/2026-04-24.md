## Resumen Técnico

### Problema Inicial
El usuario solicitó probar el proyecto para validar que el workflow de producción funcionara correctamente. 

### Decisiones Tomadas y Por Qué

1. **Pruebas en entorno aislado (real-test/)**
   - Por qué: Mantener el proyecto principal limpio durante pruebas
   - Por qué no en docker-test/: Error de tipeo - debí usar real-test/ desde el inicio

2. **Docker Compose con imagen pre-built para tasker-board**
   - Por qué: Evitar build complejo que requiere node_modules, package.json, src/, etc.
   - Alternativa considerada: Build desde código fuente pero el template no tenía todos los archivos

3. **Configuración runtime en index.html en vez de variables de entorno Docker**
   - Por qué: La imagen pre-built no soportaba variables de entorno/runtime de Vite
   - Solución: Embeber la configuración directamente en el HTML

4. **Proxy nginx con X-API-Key hardcoded**
   - Por qué: El frontend no podía pasar el API key desde el browser
   - Alternativa: Usar nginx.conf para hacer proxy y añadir el header automáticamente

5. **Uso de imagen LOCAL (socialseed-tasker-tasker-board:latest) en vez de ghcr.io**
   - Por qué: La imagen en GitHub Container Registry no estaba actualizada con los cambios necesarios
   - Nota: Esto fue necesario porque la imagen pública no tenía el nginx.conf con proxy

6. **Fix del frontend api client para manejar respuestas paginadas y no paginadas**
   - Por qué: La API a veces retornaba con paginación y a veces array directo
   - Solución: Verificar el tipo de respuesta y manejar ambos casos
